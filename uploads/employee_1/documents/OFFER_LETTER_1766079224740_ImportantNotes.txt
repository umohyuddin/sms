Why Move to Parameterized BDD
Currently, our BDD feature file contains many repeated scenarios with only small differences (channel type/expected action).
This makes it:

Harder to maintain – if a step changes, we must update it in multiple places.
Longer to read – lots of duplication makes the file harder to follow.
Error-prone – higher chance of inconsistent wording between similar cases.

Benefits of Parameterization
Reduces duplication – one Scenario Outline + Examples table can replace 30+ separate scenarios.
Centralized changes – if a step changes (e.g., text in "no match" condition), we update it once.
Easier to review – team members can quickly scan the table to see all possible test combinations.
Future-proof – adding a new match type, channel, or direction is just adding a row in the table, not rewriting a full scenario.

How It Works
We identify variables (Channel, Direction, Match Condition, Identifier, Expected Actions).
We replace hard-coded values in the steps with placeholders.
We provide all combinations in an Examples table.

I had a discussion with Hassan about this, and he suggested that if our scenarios can be grouped, we should encourage this approach.
